# MemeGPT вАФ Phase 2 Implementation: Reliability, Observability & AI Governance

Builds on top of the Phase 1 drop (RBAC, rate limiting, config validation,
debug-telemetry removal, refresh-token cookies). 18 files changed/added.

---

## 1. Fixed the blocking health check

**Changed:** `backend/routers/health.py`

`check_system_resources()` called `psutil.cpu_percent(interval=1)` directly
inside an `async def` вАФ that blocks its calling thread for a full second,
and since it's the only event loop in the process, **every other concurrent
request on that worker stalled too**, on every single `/health/detailed`
probe. Fixed three ways at once:
1. `interval=None` instead of `interval=1` вАФ non-blocking, compares against the last call instead of sleeping.
2. The still-synchronous psutil calls now run via `loop.run_in_executor()`, so they structurally can't block the event loop even if that changes.
3. Result is cached for 5 seconds, so back-to-back k8s probes don't pay even the executor round-trip repeatedly.

No deploy steps beyond the normal deploy вАФ pure code fix, no config/migration.

## 2. Observability actually wired in

**New:** `backend/core/logging.py` вАФ configures `structlog` (JSON in prod, readable console in dev), quiets noisy third-party loggers.
**New:** `backend/core/sentry.py` вАФ actually calls `sentry_sdk.init()`. This had been a declared dependency with `SENTRY_DSN` configured in every `.env*` file since day one, and nothing ever called it.
**Changed:** `backend/core/middleware.py` вАФ added `RequestIDMiddleware`: generates/propagates `X-Request-ID`, binds it into structlog's contextvars so every log line during a request carries the same ID, and tags it onto the Sentry scope.
**Changed:** `backend/core/config.py` вАФ added `sentry_dsn`, `sentry_traces_sample_rate`, `log_level`.
**Changed:** `backend/main.py` вАФ calls `configure_logging()` + `init_sentry()` before the FastAPI app is constructed.

**Deploy steps:** set `SENTRY_DSN` in your environment if you haven't already (it's likely already in your `.env.production`/`.env.staging` вАФ those files were never touched, this just finally *uses* the value). Nothing breaks if it's unset; Sentry just stays off.

## 3. Gemini вЖТ Anthropic fallback with a circuit breaker

**New:** `backend/services/circuit_breaker.py` вАФ minimal process-local breaker (closed вЖТ open after 3 consecutive failures вЖТ half-open recovery probe after 30s).
**Changed:** `backend/services/meme_ai.py`:
- Added `generate_meme_captions_with_anthropic()`, same contract as the Gemini function.
- `get_caption_generator()` now tries Gemini first, falls back to Anthropic automatically when Gemini fails or its breaker is open. Transparent to every caller вАФ `routers/ai.py` and the worker didn't need to change their calling code.
- **Tightened Gemini's safety thresholds** from `BLOCK_ONLY_HIGH` (the most permissive tier available) to `BLOCK_MEDIUM_AND_ABOVE` on all four categories. Still allows edgy meme humor through; no longer waves medium-severity content through with nothing reviewing it.

**Deploy steps:** set `ANTHROPIC_API_KEY` if you haven't (again, likely already in your env files, just unused until now). No migration needed.

**How to verify the fallback actually works:** temporarily set `GEMINI_API_KEY` to garbage in staging, fire a generation request, confirm it still succeeds and the response's `reasoning` field says "Generated by Anthropic (Gemini fallback)".

## 4. Content moderation gate before anything goes public

This is the biggest piece. Previously: generation succeeds вЖТ `is_public=True` вЖТ live on `/gallery`, instantly, no review, ever.

**New:** `backend/services/moderation.py` вАФ sends generated caption text through a dedicated classification call (Gemini preferred, Anthropic fallback вАФ it's a cheap classification call, either model handles it fine) asking a structured "is this OK for a public, general-audience meme gallery" question. Fails **open** by default (generation still succeeds, unreviewed, if the moderation call itself errors вАФ `MODERATION_FAIL_CLOSED=true` to change that once you have real traffic data on how often it happens).

**Changed:** `backend/models/models.py` вАФ `GeneratedMeme` gets `moderation_status` (`pending`/`approved`/`flagged`/`rejected`) and `moderation_reason`.
**New:** `backend/db/migrations/versions/20260628_add_moderation_fields.py` вАФ adds those columns; back-fills existing public memes to `approved` (doesn't retroactively hide already-live content, only gates new generations going forward).
**Changed:** `backend/workers/meme_worker.py` вАФ **this is the actual live code path** every real generation goes through (the `_compose_and_upload` helper in `routers/memes.py` turned out to be dead code in the original repo, never called вАФ fixed there too for consistency, but the worker is what matters). Moderation now runs concurrently with image composition (independent work, no added serial latency) and gates `is_public` + sets `moderation_status`.
**Changed:** `backend/routers/memes.py` вАФ added the admin review queue: `GET /memes/moderation/flagged`, `POST /memes/{id}/moderation/approve`, `POST /memes/{id}/moderation/reject` (all admin-only, reusing Phase 1's `get_current_admin_user`).

**Deploy steps:**
```bash
cd backend
alembic upgrade head
```
Set `MODERATION_ENABLED=true` (default) in your env. **Restart your ARQ worker process**, not just the API вАФ `meme_worker.py` changed and that's a separate process from `main.py`.

**Honest caveat, stated plainly:** this is a single LLM classification call, not a production-grade moderation vendor. It's the minimum defensible bar of "nothing unreviewed reaches a public surface automatically" вАФ good enough to close the audit finding, not a permanent solution. Evaluate a dedicated moderation API in Phase 3 if generation volume grows enough that this becomes a cost or latency concern.

## 5. Trending data honestly labeled instead of implying a live feed

**Changed:** `backend/routers/trending.py` вАФ `/topics`'s `"source": "mock"` is now `"source": "curated"`, with a docstring explaining this is hand-picked seed data, not a live Reddit/Twitter scrape. (`/topics/real`, which derives suggestions from your own users' actual prompts, was already genuinely live and is unchanged.)
**Changed:** `frontend/src/components/TrendingTopics.tsx` вАФ the footer text now reads "Curated picks to spark ideas" instead of just "Refreshes every 5 minutes" (which, next to per-topic `source: "Reddit"/"Twitter"` labels, implied a live integration that doesn't exist).
**Changed:** `frontend/src/components/Footer.tsx` вАФ the "Reddit", "Discord Bot", "Slack App" links were `href="#"` dead placeholders presented as live features. Now explicitly marked "Coming Soon" instead of looking like broken links.

**Incidental bug fix found while in this file:** `routers/trending.py`'s `get_trending_memes` was constructing `MemeResponse` without the required `prompt`, `like_count`, `is_public` fields вАФ that would raise a Pydantic validation error on every call to `GET /trending/memes`. Fixed alongside the relabeling since it was a one-line correction using data already in scope; flagging it here since it's outside this phase's stated goals.

Wiring real Reddit/NewsAPI integration (the `praw`/`newsapi-python` packages are already in `requirements.txt`) remains a backlog item вАФ it needs real API credentials provisioned, which isn't something that can be done generically here.

## 6. API versioning: `/api/v1` introduced, `/api` kept as deprecated alias

**Changed:** `backend/main.py` вАФ every router is now mounted twice: once under `/api/v1` (canonical, shows in `/docs`) and once under the legacy `/api` (kept working for any existing integrations, hidden from `/docs` via `include_in_schema=False`).
**Changed:** `backend/core/middleware.py` вАФ added `DeprecatedApiAliasMiddleware`, which tags legacy `/api/*` responses with `Deprecation: true` and a `Link` header pointing at the `/api/v1` successor.
**Changed:** `frontend/src/lib/api.ts` вАФ default base URL switched to `/api/v1`.

**вЪ†пЄП Bug I caught and fixed while doing this, worth knowing about:** the rate-limit middleware's path matching only checked `path.startswith("/api/memes/generate")` etc. Once the frontend started calling `/api/v1/memes/generate`, that check would silently stop matching вАФ **reopening the exact unbounded-generation cost exposure Phase 1 closed**, just on the new path. Added `_normalize_api_path()` in `core/middleware.py` to strip either prefix before matching, so rate limiting (and the templates/trending read limits) apply identically regardless of which version a request comes in on. Worth specifically testing this вАФ see verification checklist below.

**Deploy steps:** none beyond normal deploy. Backward-compatible вАФ anything still calling unversioned `/api/...` keeps working.

- [ ] **ARQ worker process restarted** (separate from the API process вАФ easy to miss)
- [ ] `GET /api/v1/health/detailed` hit 10x in a row in under 5s вАФ no longer takes a perceptible extra second each time
- [ ] Sentry shows a test event after deliberately raising an exception somewhere
- [ ] Logs are structured JSON (in prod) and every line for one request shares the same `request_id`
- [ ] Generate a meme with `GEMINI_API_KEY` temporarily broken in staging вЖТ still succeeds via Anthropic fallback

- [ ] **Fire 6 rapid `POST /api/v1/memes/generate/quick` calls** (note: `/api/v1`, not `/api`) in under a minute вЖТ 6th still returns `429`. This is the specific regression the path-normalization fix prevents.
- [ ] `GET /api/v1/trending/topics` response has `"source": "curated"`
- [ ] Footer shows "Coming Soon" badges on Reddit/Discord Bot/Slack App instead of dead links
